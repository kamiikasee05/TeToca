## Segunda parte: Fix horarios + migración workflows n8n

### Bug #1: Slots no se muestran en booking form
- **Síntoma:** La landing mostraba `2026-` en vez de los horarios disponibles
- **Causa raíz:** El endpoint `/availabilities` del scheduler devolvía fechas completas (`"2026-06-16 09:00:00"`) pero el frontend esperaba solo horas (`"09:00"`)
- **Fix:** `scheduler/src/routes/slots.js:114` — cambiar `toISOString().replace('T', ' ').slice(0, 19)` → `toTimeString().slice(0, 5)`
- **Resultado:** Ahora devuelve `"09:00"`, el frontend lo renderiza correctamente

### Bug #2: Todos los workflows n8n apuntaban a EasyAppointments (retirado)
- **Diagnóstico:** Los 4 workflows (WF-1 a WF-4) tenían URLs a `easyappointments:80`, credenciales HTTP Basic Auth que ya no funcionan, y nombres de campos en snake_case (EA) vs camelCase (scheduler)
- **Migración realizada:**
  - `easyappointments:80` → `scheduler:3000` (API REST) o `landing:8080` (PHP relay)
  - Auth: HTTP Basic Auth → `X-API-Key: admin2024` (vía `$env.SCHEDULER_API_KEY`)
  - `customer.phone_number` → `customer.phone`; `customer.first_name` → `customer.firstName`
  - `provider.first_name` → `provider.firstName`
- **Nuevo workflow WF-RT:** Confirmación en tiempo real vía webhook `appointment-created` del scheduler
- **Env var:** `SCHEDULER_API_KEY` agregado al servicio n8n en docker-compose.yml
