# Sesión 2026-06-18 — Deploy fixes para entorno productivo

> **⚠️ HISTÓRICO — Stack Docker Compose.** Esta sesión describe fixes para Docker Compose en WSL2/Windows. El 21 Jun 2026 se migró a bare metal. Ver [[Sesion-2026-06-21]].

**Commit:** `bdcae27` (pushed to master)

## Resumen

Sesión de deploy: se levantó el stack completo en una máquina nueva (WSL2/Docker Desktop en Windows) y se corrigieron múltiples bugs que impedían el funcionamiento en un entorno que no es el de desarrollo original.

---

## WSL2 / Docker — Port binding

**Problema:** WSL2 falla al bindear `127.0.0.1` cuando múltiples contenedores inician simultáneamente. Los contenedores no podían exponer sus puertos correctamente.

**Fix:** Cambiados todos los binds de `127.0.0.1` a `0.0.0.0` en `docker-compose.prod.yml`. Esto asegura que los puertos sean accesibles desde cualquier interfaz dentro de la VM de WSL2.

**Archivo:** `docker-compose.prod.yml`

---

## CRLF line endings — setup.sh + `source .env`

**Problema:** Git en Windows introduce terminadores CRLF (`\r\n`). Al hacer `source .env` con `set -euo pipefail`, bash interpreta el `\r` como parte del valor de la variable, rompiendo comandos como `openssl` o `php` que reciben argumentos con `\r` al final.

**Fix en `setup.sh`:**
1. Todos los valores en `.env` se envuelven en **single quotes** (`'${VAR}'`) para que bash no intente interpretar `$2y$...` (hash bcrypt) como variables.
2. Agregado `sed -i 's/\r$//' .env` después de generar el archivo para limpiar CRLF sobrantes.

```bash
# Antes (roto):
SCHEDULER_API_KEY=${SCHEDULER_API_KEY}
ADMIN_PASSWORD_HASH=${ADMIN_PASSWORD_HASH}

# Después (corregido):
SCHEDULER_API_KEY='${SCHEDULER_API_KEY}'
ADMIN_PASSWORD_HASH='${ADMIN_PASSWORD_HASH}'
sed -i 's/\r$//' .env 2>/dev/null || true
```

**Archivo:** `setup.sh`

---

## API routing — Landing SPA ↔ Scheduler

**Problema:** El SPA del landing tenía hardcodeado `http://localhost:3000/api/v1` como API base. Esto solo funcionaba en la máquina de desarrollo (localhost). En producción o en otra máquina, el browser no puede alcanzar `localhost:3000` porque el scheduler solo escucha dentro de la red Docker.

**Fix — dos cambios:**

### 1. Landing SPA: URL relativa
```javascript
// Antes (roto en prod):
var API = 'http://localhost:3000/api/v1';

// Después (funciona en cualquier entorno):
var API = '/api/v1';
```

### 2. nginx.conf: nuevo location block para `/api/v1`
```nginx
# Proxy scheduler API (NUEVO — antes no existía)
location /api/v1 {
    proxy_pass http://scheduler:3000/api/v1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}
```

**IMPORTANTE:** El bloque `/api/v1` debe ir **ANTES** del bloque `/api`. Si `/api` matchea primero, las requests a `/api/v1` van al admin PHP en vez del scheduler.

**Archivos:** `landing/index.html`, `landing/nginx.conf`

---

## Scheduler — Missing env vars

**Problema:** El contenedor `scheduler` fallaba con error FATAL al iniciar porque no recibía las variables `OPENWA_API_KEY` y `OPENWA_SESSION_ID`. El scheduler las necesita para poder proxear mensajes a OpenWA.

**Fix:** Agregadas al bloque `environment:` en `docker-compose.yml`:

```yaml
scheduler:
  environment:
    PORT: 3000
    API_KEY: ${SCHEDULER_API_KEY}
    OPENWA_API_KEY: ${OPENWA_API_KEY}        # ← NUEVO
    OPENWA_SESSION_ID: ${OPENWA_SESSION_ID}  # ← NUEVO
    N8N_WEBHOOK_URL: http://n8n:5678/webhook/
```

**Archivo:** `docker-compose.yml`

## Admin panel fixes

### saveConfig() — `is_writable()` gate removido

**Problema:** `is_writable()` retornaba `false` en bind mounts de Docker en Windows (WSL2), bloqueando todos los guardados de configuración (branding, colores, etc.).

**Fix:** Removida la validación `is_writable()`. Ahora `saveConfig()` usa `@file_put_contents()` directamente y solo verifica que el JSON encode haya sido exitoso.

```php
// Antes:
if (!is_writable($configFile)) return false;
$result = @file_put_contents($configFile, $json);

// Después:
$result = @file_put_contents($configFile, $json);
if ($result === false) return false;
```

**Archivo:** `landing-salon/admin/index.php`

### logout.php — Cookie clear + session destroy

**Problema:** El logout no era confiable a través del proxy nginx. Las sesiones quedaban colgadas.

**Fix:** `logout.php` ahora limpia explícitamente el array `$_SESSION`, destruye la cookie de sesión con parámetros correctos (path, domain, secure, httponly), y llama `session_destroy()`.

```php
$_SESSION = [];
if (ini_get("session.use_cookies")) {
    $params = session_get_cookie_params();
    setcookie(session_name(), '', time() - 42000,
        $params["path"], $params["domain"],
        $params["secure"], $params["httponly"]
    );
}
session_destroy();
```

**Archivo:** `landing-salon/admin/logout.php`

### Dockerfile — upload limits aumentados

**Problema:** Los límites por defecto de PHP (2M upload, 8M post, 15s input) eran insuficientes para subir logos e imágenes de galería.

**Fix:** Aumentados en el `CMD` del Dockerfile:

| Directiva | Antes | Después |
|---|---|---|
| `upload_max_filesize` | 2M | 20M |
| `post_max_size` | 8M | 25M |
| `max_input_time` | 15 | 60 |

```bash
CMD ["php", "-d", "max_execution_time=30", "-d", "max_input_time=60",
     "-d", "upload_max_filesize=20M", "-d", "post_max_size=25M",
     "-S", "0.0.0.0:8080", "-t", "/var/www/html"]
```

**Archivo:** `landing-salon/Dockerfile`

## Lecciones aprendidas

- **WSL2 + Docker Desktop en Windows:** siempre bindear a `0.0.0.0`, nunca a `127.0.0.1` en compose files para entornos Windows.
- **CRLF en git Windows:** todo script bash que haga `source .env` necesita `sed -i 's/\r$//'` o `dos2unix` como paso previo.
- **Single quotes en .env:** los hashes bcrypt (`$2y$...`) son interpretados como variables de bash si no están quotados. Usar siempre single quotes.
- **nginx location order importa:** prefijos más específicos (`/api/v1`) deben ir antes que prefijos más generales (`/api`).
- **SPA con API relativa:** nunca hardcodear `localhost:PORT` en el frontend. Usar paths relativos y delegar el routing al reverse proxy.
- **`is_writable()` no es confiable en Docker bind mounts** (Windows/WSL2). Usar `@file_put_contents()` con verificación del retorno.

# Security hardening — 2026-06-18 (commits `4afd018` + `edb1660`)

## Resumen

Segunda auditoría de seguridad completa post-deploy. Stack auditado en 4 componentes (scheduler, admin PHP, Docker infra, n8n/OpenWA). **11 hallazgos críticos encontrados y resueltos.** Stack queda hardened para producción.

---

## PII / Data protection

### CR-2: `GET /customers` y `GET /appointments` removidos de rutas públicas
**Riesgo:** Cualquiera podía leer la base de clientes completa (nombres, teléfonos) e historial de turnos.  
**Fix:** Ambos endpoints removidos del array `publicRoutes[]` en `scheduler/src/auth.js`. Ahora requieren `X-API-Key` header.

### CR-3: `GET /appointments/:id/cancel` ahora requiere API key
**Riesgo:** Un atacante podía cancelar turnos ajenos iterando IDs.  
**Fix:** Endpoint removido de rutas públicas.

### CR-10: Credenciales hardcodeadas en AGENTS.md
**Riesgo:** Session IDs, API keys, números de teléfono, admin password en texto plano.  
**Fix:** Reemplazadas por referencias a `.env` (`OPENWA_SESSION_ID`, `OPENWA_API_KEY`, etc.).

---

## Access control

### CR-1: `POST /api/v1/whatsapp/send` removido de rutas públicas
**Riesgo:** Cualquiera podía enviar mensajes de WhatsApp arbitrarios.  
**Fix:** Endpoint ahora requiere `X-API-Key`. n8n lo envía vía `x-api-key={{ $env.SCHEDULER_API_KEY }}`.

### CR-6: `landing-salon/api/whatsapp-send.php` eliminado
**Riesgo:** Endpoint PHP legacy sin autenticación. Código muerto.  
**Fix:** Archivo eliminado del repo. La funcionalidad está cubierta por el proxy del scheduler (autenticado).

### CR-5: Rate limiter del admin usaba `REMOTE_ADDR`
**Riesgo:** `REMOTE_ADDR` siempre era la IP del contenedor nginx. Rate limiter bloqueaba a todos los usuarios simultáneamente después de 5 intentos fallidos.  
**Fix:** Cambiado a `X-Real-IP` header (seteado por nginx en el proxy pass).

### CR-11: Webhooks scheduler→n8n ahora incluyen `X-Webhook-Token`
**Riesgo:** Cualquiera podía disparar webhooks de n8n sin autenticación (crear/cancelar turnos, enviar WhatsApp).  
**Fix:** Scheduler ahora incluye `X-Webhook-Token` header en todos los POST a n8n. n8n workflows validan el token.

---

## Infrastructure hardening

### CR-7: Nginx ahora monta `landing/config.json` (sin password)
**Riesgo:** `landing-salon/config.json` contenía el hash del password. Nginx lo servía estáticamente.  
**Fix:** El volumen de nginx ahora monta `landing/config.json` (copia limpia generada por `save-branding.php` con password stripped).

### CR-9: Admin PHP corre como non-root
**Riesgo:** Compromiso del contenedor PHP = root en el host Docker.  
**Fix:** Dockerfile del admin ahora usa `USER app`.

### CR-4: Stored XSS mitigado en admin dashboard
**Riesgo:** Datos de usuario (nombre, teléfono, servicio) se inyectaban sin escapar en templates JS del admin.  
**Fix:** Función `esc()` agregada a todos los templates JS que renderizan datos de usuario.

### Security headers en nginx.conf
Agregados headers de seguridad:
```
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```
