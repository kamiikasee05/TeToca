## WF3/WF4 — Debugging end-to-end

### Problema: Normalize node
OpenWA envuelve el payload entrante en un objeto `data`. Los workflows originales referenciaban `$json.body.from` y `$json.body.body`, pero OpenWA entrega:
```json
{ "body": { "data": { "from": "549...", "body": "CANCELAR" } } }
```
**Fix:** Cambiar referencias a `$json.body.data.from` y `$json.body.data.body`.

### Problema: Filter node
El filter original solo chequeaba el primer customer ID retornado por EA. Si el cliente tenía múltiples registros, podía fallar si el primero no matcheaba.
**Fix:** Iterar sobre todos los customer IDs retornados, usando `Number()` coercion para comparar IDs numéricos correctamente.

### Problema: Cancel node — PUT directo desde n8n
El HTTP Request node de n8n hacia PUT a EA API tenía problemas de autenticación/formato.
**Fix:** Delegar a un PHP relay que corre en el mismo servidor que EA (sin barreras de auth entre containers).

## PHP Cancel Relay

- **Archivo:** `E:\TUAHORA\tuahora\api\cancel-appointment.php`
- **Endpoint:** GET `/tuahora/api/cancel-appointment.php?id=X`
- **Funcionamiento:** Recibe `id` del appointment, hace curl PUT a EA API (`/api/v1/appointments/X`) con `{ "status": "cancelled" }`
- **Auth:** Basic auth con credenciales de EA admin
- Ver [[CancelRelay]] para documentación completa

## Docker Desktop — Estabilidad

- Docker Desktop en Windows requiere reinicios periódicos por memory leaks/inestabilidad del engine
- Los volúmenes (`openwa_data`, `n8n_data`, etc.) persisten datos entre reinicios
- Procedimiento: `docker compose down` → reiniciar Docker Desktop → `docker compose up -d`
- OpenWA reconecta automáticamente si la sesión es válida (sin re-escanear QR)
